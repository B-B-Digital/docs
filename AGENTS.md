# AGENTS.md — B&B Digital Client Docs Site

Client-facing help center for B&B Digital, built with Docusaurus in docs-only mode (no blog, no landing page — the FAQ/guides live at the site root). Polish is the canonical content language; English is a full i18n mirror.

---

## Stack

| Layer | Package | Version |
|---|---|---|
| Framework | Docusaurus | 3.10.2 (pinned exactly, not `^`) |
| Diagrams | `@docusaurus/theme-mermaid` | 3.10.2 |
| Language | TypeScript | 7.x (`tsconfig.json` does **not** extend `@docusaurus/tsconfig` — see Gotchas) |
| Package manager | pnpm | 10.x |
| Node | — | ≥20.0 |
| Deploy CLI | Wrangler | 4.x (devDependency, used only in CI/manually — not a runtime dep) |

## Commands

```bash
pnpm start                # dev server, Polish (default locale)
pnpm start --locale en    # dev server, English — dev server only ever renders ONE locale at a time
pnpm build                # production build, both locales → build/
pnpm serve                # serve the production build/ — required to test the locale dropdown
pnpm typecheck             # tsc, no emit
pnpm docusaurus write-translations --locale en   # regenerate theme translation JSON (navbar/footer/sidebar labels)
```

No test framework. CI (`.github/workflows/ci.yml`) runs `pnpm install --frozen-lockfile` then `pnpm build` on every PR and push to `main`.

Deploy (`.github/workflows/deploy.yml`) runs after CI succeeds on `main`: `wrangler deploy` to Cloudflare Workers (static assets only, `wrangler.jsonc`, no `main` script — this site has no server code). Requires `CLOUDFLARE_API_TOKEN` + `CLOUDFLARE_ACCOUNT_ID` repo secrets (a separate Cloudflare account from `BB-Digital_Portfolio` — see README "Custom domain" section).

## Content structure

Polish content lives in `docs/`, one folder per sidebar category. English is a 1:1 mirror in `i18n/en/docusaurus-plugin-content-docs/current/` — **same folder names, same filenames** (the doc ID is path-derived; renaming an English file breaks the pl↔en mapping).

```
docs/
├── start.mdx             # Hand-maintained homepage (slug: /) — hero + CategoryCard grid + popular-article tiles + contact band, in BOTH locales
├── faq/                  # "Najczęstsze pytania" — 4 docs (domain, hosting/SSL, CMS, GDPR/cookies)
├── wspolpraca/            # "Współpraca z nami" — pre-launch process (2 docs)
├── po-starcie/            # "Pierwsze kroki po starcie" — post-launch onboarding (5 docs)
├── cms/                   # "Edycja treści w CMS" — Sanity Studio guide (15 docs: 6 shared-mechanics + 9 per-content-type, ordered to match the client's actual Studio nav)
├── wsparcie/               # "Wsparcie i utrzymanie" — bug vs. Change Request, support scope, backups (4 docs)
└── marketing-i-rozwoj/     # "Marketing i rozwój strony" — domain/DNS, analytics, Google Business Profile (5 docs)
src/components/
├── CategoryCard/          # Homepage category tile — resolves href/article-count from the live sidebar (useDocsSidebar + findFirstSidebarItemLink), colored per category via a `color` prop
├── HeroBanner/            # Homepage dark hero (eyebrow + H1 + subtitle + optional search box + popular-tag pills), each tag's href resolved by docId
├── ArticleTile/           # Homepage "start here" quick-link tile (colored dot + category eyebrow + article title), resolved by docId
└── ContactBand/           # Homepage teal-gradient CTA band — `cta` is a plain visual label, not a link (no universal contact URL, see Gotchas)
src/utils/
└── sidebarLinks.ts        # findDocSidebarItem(items, docId) — shared recursive sidebar lookup used by HeroBanner and ArticleTile
src/theme/                 # Swizzled (ejected) theme-classic components, pinned to 3.10.2 — see Gotchas before touching
├── DocSidebar/Desktop/            # Adds a sticky <SidebarSearch/> above the scrolling menu (rest copied verbatim)
│   └── SidebarSearch/             # Decorative search box (⌘K) — NO backend yet, input is readOnly (see Gotchas)
└── DocSidebarItem/
    ├── Category/                  # Colored icon square + article-count badge + per-category accent CSS var on the <li>
    └── Link/                      # Renders customProps.icon (the "Start" home entry) before the label
```

`sidebars.ts` categories carry `customProps: {icon, color}` (emoji + accent hex) that the swizzled `DocSidebarItem/Category` turns into the icon square, count badge, sub-list accent rail, and active-item dot. The palette matches the homepage `<CategoryCard>` colors — keep the two in sync. The homepage doc's "Start"/"Home" sidebar entry gets its 🏠 via `sidebar_label` + `sidebar_custom_props.icon` in `start.mdx` frontmatter (both locales).

`sidebars.ts` is explicit (not autogenerated) — every new doc must be added to its category's `items` array by doc ID (folder/filename, no extension), in the order it should appear.

`docs/start.mdx` (and its English mirror) render one `<CategoryCard firstDocId="..." icon="..." title="..." description="..." />` per category rather than listing every doc — the sidebar remains the exhaustive navigation. Adding a 7th category means adding a 7th `<CategoryCard>`, not a bullet list.

Every doc's frontmatter sets a **flat root-level `slug`** (e.g. `slug: /czym-jest-cms`, not folder-derived) plus `sidebar_position` and `title`. Cross-links between docs use **file-relative Markdown links ending in `.md`** (`../faq/czym-jest-cms.md`), never absolute slug paths — relative links resolve to the correct per-locale slug automatically; absolute ones don't and silently break in English.

## Gotchas

- **`docs/start.mdx` is hand-maintained in both locales, not autogenerated.** It's a `<CategoryCard>` grid, one per category. Adding a category without adding its card (Polish AND `i18n/en/.../current/start.mdx`) leaves it undiscoverable from the homepage — `pnpm build` won't catch this.
- **Never hardcode a doc's href/slug in a custom component.** Polish and English slugs genuinely differ per doc (e.g. `/czym-jest-domena` vs. `/what-is-a-domain`) — a literal string prop would silently misroute (or 404) in one locale, and `onBrokenLinks` doesn't check arbitrary component prop strings. `CategoryCard` resolves a *category's* link via `useDocsSidebar()` + `findFirstSidebarItemLink()`; `HeroBanner` and `ArticleTile` resolve a *specific doc's* link via `findDocSidebarItem()` in `src/utils/sidebarLinks.ts` — mirror whichever pattern fits for any future component that links to a doc.
- **`useDocById()` cannot give you a doc's URL.** Its return type (`PropVersionDoc`) is `Pick<DocMetadata, 'id' | 'title' | 'description' | 'sidebar'>` — no `permalink` field, by design. To resolve a doc's href client-side you need its *sidebar link item* instead (which does carry `href`), found by walking `useDocsSidebar().items` — that's what `findDocSidebarItem()` does.
- **Client-side Docusaurus plugin packages must be added as an explicit `devDependency`**, even though they're already installed transitively via `@docusaurus/preset-classic`. pnpm's strict `node_modules` won't hoist a phantom (non-declared) dependency to a location TypeScript can resolve from source files — `@docusaurus/plugin-content-docs` (for `.../client`) and `@docusaurus/theme-common` (for the ejected sidebar components) both had to be added directly before `tsc` would resolve them, despite already existing deep in the pnpm store.
- **The `src/theme/` sidebar components are *ejected* copies pinned to theme-classic 3.10.2, not wrappers.** `DocSidebar/Desktop`, `DocSidebarItem/Category`, and `DocSidebarItem/Link` reproduce the upstream logic verbatim except for a small, commented custom region each (search box / icon+count / home icon). They do **not** auto-track upstream fixes — when bumping Docusaurus, re-diff each against the new `node_modules/.../theme-classic/lib/theme/...` source and port any changes. The matching `styles.module.css` files are copied verbatim too (the components import their class names). A category's icon/count/accent come from `sidebars.ts` `customProps`; the accent hex is set as a `--sidebar-cat-accent` CSS var on the `<li>` and consumed in `custom.css`.
- **The two search boxes (hero + sidebar) are presentational only — there is no search backend.** Both inputs are `readOnly` and wired to nothing; the ⌘K hint is decorative. They match the redesign mockup. To make search real, add a plugin (Algolia DocSearch or `@easyops-cn/docusaurus-search-local`) and replace `src/theme/DocSidebar/Desktop/SidebarSearch` + `HeroBanner`'s search markup with its `<SearchBar/>`. The sidebar placeholder is translated via `theme.SearchBar.sidebarPlaceholder` in `i18n/en/code.json`; the hero placeholder/button are per-locale props in each `start.mdx`.
- **Sidebar *category* labels need a separate i18n translation entry.** Doc titles auto-translate via each file's own frontmatter, but a category's `label` string in `sidebars.ts` only translates in the English UI if a matching key exists in `i18n/en/docusaurus-plugin-content-docs/current.json` (`"sidebar.docs.category.<Polish label>": {"message": "<English label>"}`). Miss this and the English sidebar silently shows Polish category names — build passes regardless.
- **i18n markdown fallback is silent.** A missing English translation doesn't fail the build — Docusaurus renders the Polish content under the Polish slug instead. Verify translation completeness by diffing file lists per category folder (pl vs. en) and grepping the English build output for Polish words, not just checking build success.
- **Mermaid diagrams render client-side only.** The SSR'd static HTML in `build/` contains no diagram markup at all (no `mermaid` class, no node text) even when correctly configured — the diagram mounts via JS after page load. Checking `build/**/*.html` for diagram content will always look "broken" whether or not it actually is; the only real verification is loading the page in a browser.
- **`tsconfig.json` does not extend `@docusaurus/tsconfig`.** That package's base config sets `compilerOptions.baseUrl`, which TypeScript 7 removed entirely (`TS5102`). `tsconfig.json` inlines the same options minus `baseUrl` instead — `paths` still resolves correctly relative to the tsconfig file without it. Don't re-add the `extends` line without checking the installed TypeScript major version first.
- **`wrangler.jsonc` has no `main` script on purpose.** This site is pure static output (`assets.directory: ./build`, no dynamic routes) — an asset-only Worker needs no JS entrypoint.

## Branding

Colors, fonts, and logo match `docs/branding/brand-guide.html` in `BB-Digital_Portfolio`: primary `#0c7e93` (light mode) / secondary `#22c3da` (dark mode) in `src/css/custom.css`, Space Grotesk (headings) + Inter (body) via Google Fonts.

The navbar is pinned permanently dark (`navbar.style: 'dark'` in `docusaurus.config.ts`, ink `#13171c` via `.navbar--dark` in `custom.css`) regardless of the site's light/dark color mode, matching the brand guide's own nav mockup — so there's a single logo (`static/img/logo-dark.svg`, the white mark), not a light/dark swap. Don't reintroduce `logo.srcDark`/a second logo file without also reverting `navbar.style`; the two changes go together (a mode-adaptive navbar background needs a mode-adaptive logo, a pinned-dark navbar doesn't).
